# Future Risk QA — 향후 문제가 될 구조적 위험이 있는가

현재는 동작하지만, 게임이 성장하거나 변경될 때 문제가 될 수 있는 구조적 약점을 식별한다.

---

## 우려 유형

### 1. 변경 전파 폭발 (Change Ripple)
- **관점:** 하나의 기능을 수정하면 예상치 못한 곳까지 영향이 퍼지는 구조인가?
- **살펴볼 곳:** 싱글톤을 통한 전역 상태 접근, 구체 클래스 간 직접 참조, 순환 의존
- **판단 기준:** 핵심 시스템 하나를 수정할 때 영향을 받는 파일이 5개 이상으로 추정되는 경우

### 2. 확장 저항 (Extension Resistance)
- **관점:** 새로운 콘텐츠(적 유형, 스킬, 아이템)를 추가할 때 기존 코드를 수정해야 하는가?
- **살펴볼 곳:** switch-case로 타입을 분기하는 코드, 하드코딩된 enum 분기, 팩토리 없이 직접 생성
- **판단 기준:** 새 케이스를 추가할 때마다 기존 코드의 switch/if를 수정해야 하는 구조

### 3. 단일 책임 위반 (Responsibility Overload)
- **관점:** 하나의 클래스/시스템이 너무 많은 역할을 겸하고 있어 이해와 수정이 어려운가?
- **살펴볼 곳:** 핵심 매니저 클래스의 책임 범위. "이 클래스를 한 문장으로 설명할 수 있는가?"
- **판단 기준:** 클래스의 역할을 2개 이상으로 분리할 수 있고, 각각이 독립적으로 의미 있는 경우

### 4. 기술 부채 (Technical Debt)
- **관점:** "나중에 고치자"고 남겨둔 것이 쌓여서 전체 코드 품질을 낮추고 있는가?
- **살펴볼 곳:** deprecated API 사용, 구버전 Unity API, 임시 해결(workaround)이 정착된 코드
- **판단 기준:** 기술 부채가 해당 시스템의 **유지보수 비용을 의미 있게 높이는** 경우 (사소한 것은 무시)

### 5. 테스트 불가능 구조 (Untestable Architecture)
- **관점:** 자동 테스트를 작성하기 어려운 구조인가?
- **살펴볼 곳:** 시스템 간 강한 결합, 싱글톤 의존, 구체 클래스에 대한 직접 의존, static 상태
- **판단 기준:** 핵심 로직을 모킹/격리 없이 단위 테스트할 수 없는 경우

### 6. 스케일링 한계 (Scaling Limit)
- **관점:** 게임 규모(플레이어 수, 데이터 양, 콘텐츠 양)가 커지면 깨지는 설계가 있는가?
- **살펴볼 곳:** 데이터를 전부 메모리에 올리는 구조, 선형 탐색으로 대량 데이터 처리, 한 씬에 모든 오브젝트 배치
- **판단 기준:** 현재 규모에서는 동작하지만, 콘텐츠가 2-3배 늘어나면 문제가 될 구조

---

## 코드 검증 관점

- 핵심 시스템의 **의존성 그래프**를 파악하여 변경 영향 범위 추정
- "새 콘텐츠 추가" 시나리오를 시뮬레이션하여 **수정 필요 파일 수** 추정
- 매니저 클래스의 **public API 수와 책임 범위** 분석
- deprecated API의 **대체 API 존재 여부**와 마이그레이션 난이도 평가

---

## 런타임 검증 관점

이 모듈은 일반적으로 런타임 검증이 불필요하다 (구조 문제는 정적 분석으로 충분).

예외:
- 싱글톤 인스턴스 중복 생성 여부 확인
- 대량 데이터 로드 시 메모리 사용량 측정
- 콘텐츠 증가 시뮬레이션(대량 오브젝트 생성)으로 스케일링 한계 확인
